## Context

当前应用使用硬编码的暗色主题，所有样式直接写在组件中。用户需要在不同环境下切换主题以获得更好的使用体验。

**当前状态：**
- Tailwind CSS 4.x 已配置暗色模式
- shadcn/ui 组件使用 CSS 变量（`--background`, `--foreground` 等）
- 没有集中的主题管理机制

**约束：**
- 必须支持后续扩展更多主题（护眼模式、高对比度等）
- 不能破坏现有 UI 样式
- 需要与 Electron 的 nativeTheme API 集成

## Goals / Non-Goals

**Goals:**
- 实现可扩展的主题注册机制
- 支持暗色/亮色两种内置主题
- 支持系统主题自动跟随
- 持久化用户主题选择
- 所有组件通过 CSS 变量获取颜色

**Non-Goals:**
- 不实现自定义主题编辑器
- 不支持用户上传自定义主题
- 不实现主题动画过渡效果（可后续添加）

## Decisions

### 1. 主题架构：注册制 + CSS 变量

**决策**：使用主题注册机制，每个主题定义一套设计令牌，运行时动态注入 CSS 变量。

**理由**：
- 扩展性强：添加新主题只需注册新的主题对象
- 性能好：CSS 变量原生支持，无需运行时计算
- 兼容性：与 shadcn/ui 的 CSS 变量系统完美兼容

**替代方案**：
- Tailwind darkMode 类切换：不够灵活，难以支持多主题
- CSS-in-JS：增加运行时开销，与现有 Tailwind 方案冲突

### 2. 主题存储：electron-store + IPC

**决策**：主题配置存储在主进程的 electron-store 中，通过 IPC 同步到渲染进程。

**理由**：
- 与现有设置存储方案一致（API Key 等已使用 electron-store）
- 主进程可以原生访问 nativeTheme API
- 单一数据源，避免主进程/渲染进程状态不一致

**替代方案**：
- localStorage：无法访问 nativeTheme，与现有存储方案不一致
- 配置文件：不便于运行时修改

### 3. 主题切换时机：立即生效 + 系统跟随

**决策**：支持三种模式：
- `light`：强制亮色
- `dark`：强制暗色
- `system`：跟随系统（默认）

**理由**：
- 满足不同用户偏好
- 系统跟随作为默认值，提供开箱即用的体验
- 与 Electron nativeTheme 的三种主题源对应

### 4. 设计令牌层级

**决策**：定义三层设计令牌：
1. **基础令牌**：颜色、间距、字体等原始值
2. **语义令牌**：`--primary`, `--background`, `--foreground` 等
3. **组件令牌**：特定组件的样式变量（如需要）

**理由**：
- 基础令牌保证一致性
- 语义令牌提供可读性
- 组件令牌处理特殊情况

## Risks / Trade-offs

**[风险] 组件样式兼容性** → 逐步迁移，先确保核心组件支持，标记不兼容组件

**[风险] 主题切换闪烁** → 在 `<head>` 中注入初始化脚本，避免 FOUC（Flash of Unstyled Content）

**[风险] 第三方组件主题不一致** → 优先使用 shadcn/ui 组件，第三方组件通过 CSS 变量覆盖

**[权衡] CSS 变量 vs Tailwind 类** → 选择 CSS 变量，因为更灵活且与 shadcn/ui 兼容，但需要维护两套颜色定义

## Migration Plan

1. **Phase 1：基础设施**
   - 创建主题定义和注册机制
   - 配置 CSS 变量注入
   - 创建 theme-store 和 IPC 通信

2. **Phase 2：核心组件**
   - 更新 shadcn/ui 组件的 CSS 变量
   - 更新 AppShell、StatusBar 等核心组件
   - 添加主题切换 UI

3. **Phase 3：全量迁移**
   - 更新所有页面和组件
   - 测试所有主题下的视觉效果

**回滚策略**：保留原有样式作为 fallback，如果新主题系统出现问题可以快速回退。
