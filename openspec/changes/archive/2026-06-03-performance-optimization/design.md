## Context

当前应用使用简单的列表渲染方式，所有消息/节点都一次性渲染。随着功能增加，性能问题逐渐显现。

**当前状态：**
- 消息列表使用 `map()` 渲染所有消息
- 工作流画布使用 @xyflow/react 默认配置
- 没有代码分割，所有功能模块同步加载
- 会话数据无清理机制

**约束：**
- 不能破坏现有功能
- 需要支持后续扩展
- 保持代码可维护性

## Goals / Non-Goals

**Goals:**
- 消息列表支持万级消息流畅滚动
- 首屏加载时间减少 50%
- 内存占用稳定，不随使用时间增长
- 保持现有功能完整性

**Non-Goals:**
- 不实现服务端渲染（SSR）
- 不更换状态管理库
- 不重写核心业务逻辑

## Decisions

### 1. 虚拟滚动方案：react-window

**决策**：使用 react-window 实现消息列表虚拟滚动

**理由**：
- 成熟稳定，社区活跃
- 与 React 集成良好
- 支持动态高度（使用 VariableSizeList）
- 包体积小（~3KB gzipped）

**替代方案**：
- react-virtualized：功能更全但体积更大
- @tanstack/react-virtual：更灵活但学习成本高
- 自实现：开发成本高，维护困难

### 2. 代码分割方案：React.lazy + Suspense

**决策**：使用 React.lazy 和 Suspense 实现路由级别代码分割

**理由**：
- React 原生支持，无需额外依赖
- 与 Vite 的动态导入完美配合
- 实现简单，维护成本低

**替代方案**：
- @loadable/component：更灵活但增加依赖
- 手动动态导入：代码复杂度高

### 3. 内存管理策略

**决策**：实现会话数据生命周期管理
- 限制每个会话的消息数量（最大 1000 条）
- 组件卸载时清理事件监听器和定时器
- 使用 WeakMap 存储临时数据

**理由**：
- 防止内存泄漏
- 保持性能稳定
- 与现有架构兼容

### 4. 渲染优化策略

**决策**：关键组件使用 React.memo + useMemo/useCallback
- MessageBubble 使用 React.memo
- 消息列表使用 useMemo 缓存计算结果
- 事件处理函数使用 useCallback

**理由**：
- 减少不必要重渲染
- 提升交互响应速度
- 改动范围小，风险低

## Risks / Trade-offs

**[风险] 虚拟滚动与动态高度** → 使用 VariableSizeList + 高度预估，允许运行时调整

**[风险] 代码分割导致加载闪烁** → 使用 Suspense + Loading 状态，优化用户体验

**[风险] 内存清理可能导致数据丢失** → 只清理已持久化的数据，保留未保存内容

**[权衡] 包体积 vs 功能** → 优先使用轻量级库，必要时牺牲部分功能

## Migration Plan

1. **Phase 1：虚拟滚动**
   - 实现 MessageList 虚拟滚动
   - 测试各种消息类型（文本、代码、工具调用）
   - 验证滚动行为和性能

2. **Phase 2：代码分割**
   - 配置 Vite 代码分割
   - 添加路由级别懒加载
   - 测试加载行为

3. **Phase 3：内存优化**
   - 实现会话数据清理
   - 添加内存监控
   - 压力测试
