## 上下文

当前 Agent 工具执行流程：渲染进程 → IPC → 主进程 `AgentSessionManager` → pi-coding-agent SDK → 工具执行。工具执行无任何权限检查，Agent 可直接调用 bash、read/write/edit 等工具。

相关模块：
- `src/main/agent/session-manager.ts` — Agent 会话管理
- `src/main/agent/tool-registry.ts` — 工具注册
- `src/main/sandbox/` — 代码沙箱（Docker/ChildProcess）
- `src/preload/index.ts` — IPC 桥接
- `src/renderer/src/stores/` — 状态管理

## 目标 / 非目标

**目标：**
- 工具执行前可配置用户确认模式
- 文件路径访问控制（工作目录白名单、敏感路径黑名单）
- 危险 shell 命令拦截
- 工具级别启用/禁用
- 工具调用日志审计

**非目标：**
- 多用户权限系统（当前是单用户桌面应用）
- 远程审计日志上报
- 工具执行沙箱改造（已有 Docker 沙箱）
- Agent 自身的 prompt 注入防护

## 决策

### 1. 权限检查层位置：主进程工具注册层

**选择**：在 `ToolRegistry` 和工具执行之间插入权限检查中间件。

**替代方案**：
- A. 在 SDK 层拦截 → 侵入第三方 SDK，升级困难
- B. 在渲染进程拦截 → 可被绕过，不安全
- C. 在 IPC 层拦截 → 需要解析工具参数，耦合度高

**理由**：主进程是安全边界，在工具实际执行前拦截最可靠。通过装饰器模式包装原始工具，无需修改 SDK。

### 2. 确认机制：IPC 请求-响应 + 渲染进程弹窗

**选择**：主进程发送确认请求到渲染进程，渲染进程弹窗等待用户响应，通过 IPC 返回结果。

**替代方案**：
- A. 主进程原生 dialog → 无法自定义 UI，体验差
- B. 纯配置无确认 → 无法应对动态场景

**理由**：利用已有的 IPC 通道和 UI 组件，用户体验一致。使用 `ipcMain.handle` / `ipcRenderer.invoke` 实现同步等待。

### 3. 日志存储：electron-store + JSON Lines

**选择**：使用 `electron-store` 按会话存储工具调用日志，JSON Lines 格式。

**替代方案**：
- A. SQLite → 增加依赖，过度设计
- B. 纯内存 → 重启丢失
- C. ChromaDB → 向量数据库不适合结构化日志

**理由**：项目已使用 `electron-store`，无需新增依赖。JSON Lines 便于追加写入和按行解析。

### 4. 危险命令检测：正则规则引擎

**选择**：预定义正则规则列表，匹配 shell 命令中的危险模式。

**替代方案**：
- A. AST 解析 shell → 复杂度高，边界情况多
- B. LLM 判断 → 延迟高，不可靠

**理由**：正则规则简单高效，覆盖 `rm -rf`、`sudo`、`chmod 777`、`curl | bash` 等常见危险模式。规则可配置，用户可自定义。

### 5. 路径访问控制：基于 `process.cwd()` 的白名单

**选择**：默认允许访问当前工作目录，用户可配置额外允许路径和敏感路径黑名单。

**理由**：与 pi-coding-agent 的 `cwd` 概念一致。敏感路径（`~/.ssh`、`~/.gnupg`）硬编码默认值，用户可扩展。

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| 频繁弹窗影响体验 | 提供 `auto`/`smart`/`prompt` 三档，`smart` 模式仅高风险弹窗 |
| 权限检查增加延迟 | 检查逻辑在主进程同步执行，路径检查和正则匹配 < 1ms |
| 正则规则误判 | 规则可配置，提供白名单机制，用户可跳过特定规则 |
| 会话级记住选择可能导致意外放行 | 仅当前会话有效，重启后重置 |
| 日志文件增长 | 30 天自动清理，支持手动清理 |
