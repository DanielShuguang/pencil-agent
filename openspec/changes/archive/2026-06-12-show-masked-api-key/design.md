## Context

当前 API Key 管理界面在保存密钥后只显示"已保存"文本，用户无法确认已保存的密钥是否正确。这可能导致用户在配置多个 provider 时混淆，或在需要验证时无法快速识别已保存的密钥。改进此功能可以提升用户体验和配置准确性。

## Goals / Non-Goals

**Goals:**
- 在 ApiKeyForm 和 ProviderForm 中显示加密格式的 API Key
- 使用安全的方式，前端不接触实际 API Key
- 保持现有删除和编辑功能
- 提供清晰的错误处理

**Non-Goals:**
- 不改变 API Key 的存储机制
- 不添加新的 provider 类型
- 不修改现有的加密/解密逻辑

## Decisions

### 决策 1：在主进程中实现加密显示逻辑

**选择**：添加新的 IPC 处理程序 `settings:get-masked-key`，在主进程中返回加密的 API Key

**理由**：
- 安全性：实际 API Key 不会暴露给前端
- 性能：避免在前端进行不必要的解密操作
- 一致性：复用现有的 `safeStorage` 解密逻辑

**替代方案**：
- 在前端解密并加密显示：安全性较低，实际 API Key 会暴露给前端
- 存储时同时存储加密版本：需要修改存储结构，增加复杂性

### 决策 2：加密显示格式

**选择**：前4字符 + `***` + 后4字符（如 `sk-a***b1c2`）

**理由**：
- 提供足够的信息用于验证（前缀和后缀）
- 保护中间部分的敏感信息
- 格式简洁，易于识别

**替代方案**：
- 完全用 `*` 替代：无法验证密钥
- 显示更多字符：安全性降低

### 决策 3：ApiKeyForm 中不允许直接编辑

**选择**：在 ApiKeyForm 中显示加密 key，但不允许直接编辑，只能删除后重新输入

**理由**：
- 简化用户界面
- 避免用户误操作
- 保持现有的删除-重新输入工作流

**替代方案**：
- 允许直接编辑：增加复杂性，可能引起混淆

### 决策 4：ProviderForm 中允许直接修改

**选择**：在 ProviderForm 中显示加密 key，并允许用户直接修改

**理由**：
- ProviderForm 用于编辑 provider 配置，包括 API Key
- 用户可能需要更新 API Key
- 保持表单的一致性

**替代方案**：
- 不允许修改：需要删除后重新创建 provider，用户体验差

## Risks / Trade-offs

### 风险 1：加密显示可能泄露部分信息
- **风险**：显示前4和后4字符可能泄露部分 API Key 信息
- **缓解**：这是安全性和可用性的权衡，显示的信息有限，且 API Key 本身有固定的格式前缀

### 风险 2：主进程性能影响
- **风险**：每次显示都需要主进程解密和加密
- **缓解**：API Key 通常很短，性能影响可以忽略不计

### 风险 3：错误处理复杂性
- **风险**：解密失败、key 不存在等情况需要妥善处理
- **缓解**：实现完善的错误处理和用户反馈机制
